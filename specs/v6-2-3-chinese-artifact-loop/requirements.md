# v6.2-v6.3 中文主入口与产物闭环需求

## 目标

Ultimate PPT Master 应让中文办公用户在不理解内部脚本和合同的前提下，从任务与资料出发，确认故事板，交给本地 Agent 生产，并在工作台下载经过明确质量标记的真实产物。

## 用户故事

1. 作为首次访问者，我能在 GitHub 和 Pages 首屏立即看懂产品、案例与安装方式。
2. 作为制作 PPT 的用户，我确认过的故事板必须成为最终生产合同，而不是在 Bridge 中被重建。
3. 作为本地用户，我能从工作台启动或复制 Agent 命令，并在生成后直接下载产物。
4. 作为谨慎的办公用户，我能区分结构稿、待复核产物和已验证产物。

## 验收标准

- 当用户修改任意页面标题、结论或结构方案并创建 handoff 时，系统应在 `storyboard.json` 中保留相同 `slideId`、顺序与用户选择。
- 当请求包含 4 到 24 页时，系统应生成对应数量的稳定 `PNN` 页面，不截断为 12 页。
- 当 Bridge 检测到允许目录内的新产物时，工作台应在下一次轮询内显示并允许下载。
- 当产物尚未通过质量检查时，系统应允许下载但不得标记为已交付。
- 当请求路径离开 Bridge `outputDir`、经过外部符号链接或指向源资料时，Bridge 应拒绝访问和 Agent 启动。
- 当 GitHub 打开仓库首页时，应默认显示中文 `README.md`；英文完整镜像应位于 `README.en.md`。
- 当访客打开案例库时，应首先看到一个可编辑正式办公 PPTX 和一个完整 AI Web Deck，并可检查输入、输出和质量证据。

## 非目标

- 不建设账号、云端生成、托管上传、队列、数据库或遥测。
- 不新增 Provider、通用幻灯片编辑器或新的预设数量。
- 不重设计 Desktop；只保持共享合同兼容。
- 不公开真实交行业务材料。
