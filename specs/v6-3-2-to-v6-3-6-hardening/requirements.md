# v6.3.2–v6.3.6 体验、安全与交付真实性需求

## 1. 目标

在不增加云端后台、账号、数据库、遥测、模型 Provider 或大而全编辑器的前提下，把 v6 工作台收口为一个轻量、可信、好看的本地生产闭环：任务输入清楚，资料边界真实，故事板修改可追踪，Agent 产物可恢复，最终文件可验证下载。

本轮同时纠正视觉方向：默认首页与演示封面采用暖白、浅灰或暖纸主表面；墨黑只承担文字、细线与极少数经场景明确授权的深色页面。普通界面和正式办公演示使用柔和圆角，不再让大面积硬直角成为默认气质。

## 2. 范围

### 2.1 包含

- Web v6 工作台、案例库与公开视觉资产。
- Bridge 的 handoff、附件、SSE、Agent 启动、产物发现和下载接口。
- `DeckSession`、`DeckIR 1.0`、Best-Effect 与现有 handoff 的兼容收口。
- Desktop 的合同兼容、Rust lint 与必要安全边界；不扩展桌面功能。
- 中文 README、文档、Proof Pack、版本号与发布门禁。

### 2.2 不包含

- 云端文件托管、用户账号、数据库和远程任务队列。
- 新模型 Provider、新预设数量和完整在线编辑器。
- 直接复制 `gzh-design-skill` 的 AGPL-3.0 代码或组件实现。
- 将 Classic 控制台继续演进；它只保留兼容入口。

## 3. 功能与质量需求

### R1 默认视觉必须轻盈、圆润且内容优先

1. 当用户打开工作台、案例库或正式办公案例时，系统必须以暖白/浅灰/暖纸作为主要页面表面，`#171714` 只作为文字、细规则或小面积强调色。
2. 当系统自动推荐或自动生成封面时，正式办公、咨询、培训、自动选题和普通产品介绍必须使用浅色封面；不得自动生成满版黑色封面。
3. 只有用户明确选择深色方向，或真实品牌/影像素材需要深色舞台时，系统才可以使用深色整页，并必须在方向说明中写明原因。
4. 普通面板必须使用 16–24px 的柔和圆角，输入控件与按钮使用 10–14px 圆角；状态标签可以使用胶囊形。图表坐标、表格基线和证据规则线仍可保持直线。
5. 页面不得用重复卡片墙制造层级。每个阶段只保留一个主操作，次要操作使用浅底、描边或文本链接。
6. 品牌色只用于少数锚点；页面阅读应主要由白底、灰阶、留白、字号和结构承担。
7. 工作台和公开案例页必须在 1440px 与 390px 下无横向溢出，任务输入在首屏可见。

### R2 资料输入必须可恢复、可限制且不碰撞

1. 当用户选择同名文件时，系统必须为每份资料生成不同的稳定 source id，移除一份不得影响另一份。
2. Web 必须在读取二进制内容前检查单文件大小、文件数量与累计大小；超限时给出可恢复的中文行内错误，不得先转为 base64 再失败。
3. Bridge 必须验证附件声明大小与实际解码大小，并限制附件数量、单文件大小和累计大小。
4. 刷新后，URL 资料可以恢复；浏览器不可持久化的本地文件必须明确标记“需重新选择”，且不得继续贡献生产就绪或 grounded 状态。
5. 清洗后同名的附件必须得到唯一磁盘名，不得覆盖已有文件。
6. 超过请求体上限时，Bridge 必须稳定返回 `413` JSON 响应，不得通过主动销毁连接制造不透明的网络错误。

### R3 任务上下文与事实证据必须严格分离

1. 当用户只填写任务、受众和核心信息而未提供真实资料时，系统必须保持 `readyForProduction=false`，所有需要事实支持的页面不得标为 `grounded`。
2. `taskContext` 只用于策划与约束，不得被 Bridge 当作来源 claim。
3. URL 只有在 Bridge 成功抓取并记录 provenance 后才可计入真实证据；抓取中或失败时必须是 `pending`、`partial` 或 `missing`。
4. 用户修改的页面顺序、`slideId`、标题、结论、角色和 variant 必须逐字段进入 `storyboard.json` 与 DeckIR。
5. 单页修订必须验证 `slideId` 与 variant 属于当前故事板；未知页面或方案必须拒绝，并保留可追溯的修订历史。

### R4 Bridge 的变更型接口必须具备本地安全边界

1. 当请求携带不允许的浏览器 `Origin` 时，所有变更型接口必须在执行副作用前返回 `403`；仅省略 CORS 响应头不算拒绝。
2. 所有 JSON POST 接口必须要求 `Content-Type: application/json`，不兼容类型返回 `415`。
3. 所有项目内写入必须经过同一个安全路径解析器；绝对路径、反斜杠、空段、`.`、`..`、外部符号链接和解析后越界必须被拒绝。
4. Handoff 项目目录必须使用不可碰撞的随机后缀或 `mkdtemp`；同标题高并发不得共享目录或覆盖内容。
5. `POST /agent/launch`、单页修订、artifact 列表与下载必须使用同一项目身份校验。
6. 未启用自动启动时只返回可复制命令；启用后必须避免同一项目重复启动，并区分“已发起”与“Agent 正在运行/已完成”。

### R5 SSE、轮询和产物交付必须属于正确任务

1. 每次 handoff 必须有不可猜测或足够唯一的 `sessionId/requestId`；SSE 事件必须按该标识订阅或由 Web 严格过滤。
2. 任何标签页不得把另一个任务的 `projectPath` 写入当前 session。
3. 产物轮询必须单飞，旧请求不得覆盖新状态；页面隐藏时暂停，交付后停止。
4. 只有 `pptx | web-deck | pdf | archive` 中至少一份真实成品为 `passed`，且全部页面获批时，才允许标记“已交付”。`report` 自身不得充当最终成品。
5. 零字节、`.partial`、仍在写入或超出扫描白名单/深度/数量上限的文件不得作为可下载成品。
6. 质量报告必须能绑定到具体产物路径与摘要；旧报告不得自动使新文件变为 passed。

### R6 交互与可访问性必须可用而非仅通过静态检查

1. 故事板 variant 与用途选择的 radio 组必须支持方向键、Home/End 和 roving tabindex。
2. 诊断对话框必须捕获焦点、Escape 关闭、关闭后把焦点还给触发按钮。
3. 系统在 `prefers-reduced-motion: reduce` 下不得执行平滑滚动或非必要动画。
4. Bridge 离线、无 Agent、可启动、生成中、成品完成和质量阻断六种状态必须各有唯一主操作和可理解的中文状态。
5. 所有可恢复错误必须在对应控件附近呈现，不得只写入控制台。
6. Classic 返回链接必须尊重 GitHub Pages 的 `BASE_URL`。

### R7 版本与公开 Proof 必须诚实一致

1. 根包、Web、Desktop、Cargo 与 Tauri 版本必须一致推进至目标版本。
2. v6.3.2–v6.3.6 每个版本必须有中文主发布说明和英文镜像，并明确该版本的可独立回滚边界。
3. README、案例库和公开报告不得把内部质量合同包装成第三方 benchmark。
4. 未完成的 WPS/PowerPoint/LibreOffice 人工复核必须保持 warning/pending，不得通过沿用旧版本结论改为 passed。
5. Pages 只能从通过 CI 的 `main` 部署；未提交、未推送或未部署时不得声称已发布。

## 4. 非功能需求

- Web 主入口 JS gzip 必须保持在 80KB 以内；Classic 继续异步分包。
- Node、Python、Rust、Web 与 Desktop 构建必须可重复运行。
- Bridge 默认只绑定回环地址。
- 浏览器集成测试必须覆盖 1440px、390px、键盘与刷新恢复。
- 关键安全回归必须以可执行测试证明，不以源代码字符串搜索代替。

## 5. 验收场景

1. **无资料任务**：只填任务并生成 4 页故事板，所有事实页仍为缺失/部分，生产被阻断。
2. **同名资料**：上传两个 `report.md`，两份均保留、可独立删除，Bridge 磁盘文件不覆盖。
3. **恶意浏览器**：`Origin: https://evil.example` + `text/plain` POST 返回 403/415，磁盘无新增项目。
4. **路径越界**：`prompts/../../escaped.md` 被拒绝，outputDir 以外无文件。
5. **并发 handoff**：120 个同标题请求得到 120 个不同项目目录，内容互不覆盖。
6. **多标签页**：A 创建项目时 B 不更新 projectPath。
7. **报告不等于成品**：只有 passed 质量报告时，交付按钮仍禁用。
8. **浅色默认封面**：自动生成的正式办公、咨询、培训与普通产品封面均为浅色；深色只能由明确选择触发。
9. **圆润视觉**：工作台、案例主卡、证据面板和封面主面板符合圆角尺度，图表规则线保持精确直线。
10. **移动端**：390px 下任务输入、上传与唯一主操作在自然滚动内可达，无裁切和横向溢出。
